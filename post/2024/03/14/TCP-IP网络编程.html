1 理解网络编程与套接字

1.1 常用函数

  • 套接字socket

    • 俩台计算机直接进行数据传输需要物理连接（互联网）与传输数据软件。
    • 套接字就是网络数据传输用的软件设备，即编程中进行网络连接的工具。

  • bind

    • 每一个套接字应该绑定一个地址，别人才能根据分配的地址建立连接。

  • listen

    • 将套接字设置为可接受连接状态

  • accept

    • 受理请求连接的套接字

1.2 套接字创建过程

  • 服务端
  1. 调用socket函数创建套接字。
  2. 调用bind函数跟配ip地址和端口号。
  3. 调用listen函数转为可接受请求状态。
  4. 调用acccept函数受理连接请求。
  • 客户端
  1. 调用socket函数创建套接字。
  2. 调用connect函数想服务器端发送连接请求。

2 套接字类型与协议设置

  • 协议是为了完成数据交换而定好的约定
    • domain:协议族信息
    • type:套接字类型
    • protocol：计算机通信使用的协议信息

2.1 协议族

  • 套接字通信的一些分类
    • PF_INET:ipv4互联网协议族
    • PF_INET6:ipv6互联网协议族
    • PF_LOCAL:本地通信的UNIX协议族
    • PF_PACKET:底层套接字的协议族
    • PF_IPX:IPX Novell协议族

2.2 套接字类型

  • 套接字的数据传输方式

2.2.1 面向连接的套接字 SOCK_STREAM

  • 传输过程中数据不会丢失
  • 按序传输数据
  • 传输的数据不存在数据边界
    • 解释：套接字内部由一个由字节数组构成的缓冲，无论服务端write了几次，接收端可以立即read，也可以等缓冲满了再read
    • ps：若缓冲满了接收端仍然不read，传输段套接字将停止传输。面向连接的套接字会根据接收端的状态传输数据，还提供重传服务
  • 套接字连接必须一一对应

2.2.2面向消息的套接字 SOCK_DGRAM

  • 强调快速传输而非传输顺序
  • 传输的而数据有可能丢失也有可能损毁
  • 传输的数据有数据边界
  • 限制每次传输的数据大小

2.3 协议的最终选择

  • 一般socket()的前俩个函数（协议族、套接字类型）已经可以决定采用的协议，此时此参数直接传0
  • 若同一协议族中存在多个数据传输方式相同的协议，需要通过此参数指定具体协议信息。

3 地址族与数据序列

3.1 分配给套接字的IP地址与端口号

3.1.1 网络地址

  • IP地址分为俩类
    • IPV4 4字节地址族
    • IPV6 16字节地址族
  • IPV4地址分类
    • A类地址：首字节0~127
    • B类地址：首字节128~191
    • C类地址：首字节192~223

3.1.2 端口号

  • 端口号用于区分不同套接字，如操作系统同时创建了俩个套接字（如看视频、开网页），此时接收的数据通过数据自带的端口号分配给不同对应的套接字
  • 端口号由16位构成（065535）。其中01023是知名端口号，分配给特定应用程序，应分配此范围之外的值
  • 端口号不能重复，但TCP与UDP套接字不会共用端口号，因此可以创建端口号相同的TCP、UDP套接字

数据传输的目标地址必须同时包含IP地址与端口号，才能被传输到最终的目的应用程序套接字

3.3 网络字节序与地址变换

3.3.1 字节序

  • 大端序（网络字节序）
    • 高位字节存放到低位地址
    • 如在0x20地址开始存放数据0x12345678，则
      地址： 0x20 0x21 0x22 0x23
      数据： 0x12 0x34 0x56 0x78
  • 小端序（主流）
    • 高位字节存放到高位地址
    • 如在0x20地址开始存放数据0x12345678，则
      地址： 0x20 0x21 0x22 0x23
      数据： 0x78 0x56 0x34 0x12

不同字节序互相传输数据可能会导致数据异常，如大端序向小端序传递0x1234，小端序会识别为0x3412
因此在网络数据传输过程中，同一转为为网络字节序（大端序），再进行传输

3.3.2 字节序转换

  • 函数列表
    • unsigned short htons
    • unsigned short ntohs
    • unsigned long htonl
    • unsigned long ntohl

      通过函数名理解函数功能，其中n为network（网络字节序），h为host（主机字节序）

4 基于TCP的服务器端/客户端

4.1 TCP/IP协议栈

4.1.1 OSI七层网络模型

  • 物理层
  • 数据链路层
  • 网络层
  • 传输层
  • 会话层
  • 表示层
  • 应用层

4.1.2 TCP/IP协议栈

  • 链路层
    • 俩台主机直接物理连接的网络标准，如WAN、LAN等
  • IP层
    • 传输数据时的路径选择协议
  • TCP/UDP层（传输层）
    • 以IP层的路径选择为基础，根据选择的协议完成实际的数据传输
  • 应用层
    • 根据应用程序的特点完成服务器端和客户端之间的数据传输规则

4.2 实现基于TCP的服务器端/客户端

4.2.1 进入等待连接请求状态（listen）

  • 参数1 sock
    • 用于处理客户端发起的连接请求的套接字（直接用服务端套接字自己），类似于门卫的角色
    • 若客户端发起的连接请求被通过，则会进入”等候室”，等待服务端进行连接交互（accept）
  • 参数2 backlog
    • 决定了等候室大小
    • 与服务器端的特性有关，频繁接收请求的Web服务器端至少为15，具体根据实验结果确定

只有服务端调用了listen函数，客户端才能正常调用connect函数（进入等候室），否则会报错

4.2.2 受理客户端连接请求

成功时返回创建的套接字文件描述符，失败返回-1
accept函数内部会自动产生用于数据I/O的套接字，该套接字自动与发起连接请求的客户端建立连接

5 TCP原理

  • TCP套接字的数据收发无边界
    • write函数调用瞬间，数据移到输出缓冲，read函数调用瞬间，从输入缓冲读取数据
    • 接收端 - 发送端
      write -> 输出缓冲 -> 输入缓冲 -> read
      read <- 输入缓冲 <- 输出缓冲 <- write
  • 缓冲特性
    • I/O缓冲在每个TCP套接字中单独存在
    • I/O缓冲在创建套接字时自动生成
    • 即使关闭套接字也会继续传递输出缓冲中遗留的数据
    • 关闭套接字将丢失输入缓冲中的数据
  • 三次握手

    完成数据交换准备

    • 套接字以全双工方式工作，可以双向传递数据
    • SEQ + YNC
      • TCP在连接过程中发送数据包时需分配序号
      • 主机a -> 主机b SEQ:1000 ACK:-
      • 主机b -> 主机a SEQ:2000 ACK:1001
      • 主机a -> 主机b SEQ:1001 ACK:2001
      • 通过向数据包分配序号并确认，可以在数据丢失时马上查看并重传丢失的数据包
  • 数据交换
    • 主机a -> 主机b SEQ:1000 100byte
    • 主机b -> 主机a ACK:1101

      接收端返回下一待传递数据包序号 若发送端接收响应超时或丢失则重传

  • 四次握手

    断开连接

    • 主机a -> 主机b FIN SEQ:5000 ACK:-
    • 主机b -> 主机a ACK SEQ:7500 ACK:5001
    • 主机b -> 主机a FIN SEQ:7501 ACK:5001
    • 主机a -> 主机b ACK SEQ:5001 ACK:7502
    • A向B传递断开连接的消息，B发送确认收到的响应，B再向A传递断开连接的消息，A再向B发送确认收到的响应

6 UDP

  • 与TCP不同，UDP不会进行流控制
  • TCP比UDP慢的原因
    • 收发数据过程中为了保证可靠性而添加的流控制
    • 收发数据前后进行的连接设置及清除过程
  • 如果收发的数据量小但需要频繁连接时，UDP比TCP更高效

6.1 基于UDP的服务端/客户端

  • UDP无需像TCP那样在连接状态下交换数据，因此无需调用listen和accept函数
  • UDP服务器端和客户端均只需要一个套接字，不想TCP是一一对应的，10个客户端套接字就需要10个服务端套接字
